## 1. API 개념 정리

### API (Application Programming Interface)란?
- **"프로그램끼리 대화하는 약속된 방식"**
- 핵심: 내부 구현을 몰라도 정해진 규칙으로 기능을 사용할 수 있게 한 인터페이스
- 컴퓨터 과학에서는 이를 **추상화(Abstraction)**라고 함

### API를 만드는 이유 (개발자 관점)
| 목적 | 예시 |
|------|------|
| **수익 창출** | Gemini, OpenAI — API 호출량에 따라 과금 (SaaS 모델) |
| **생태계 확장** | 공공데이터포털 — 민간이 데이터를 활용해 서비스를 만들도록 개방 |
| **구조적 분리** | 위니쿠키 FastAPI — 프론트/백을 독립시켜 유지보수와 확장을 쉽게 함 |

→ 구조적 분리의 핵심 원칙: **관심사의 분리 (Separation of Concerns)**
  - 프론트엔드(Discord → Streamlit)를 바꿔도 백엔드는 손대지 않아도 됨

## 2. REST API

### REST (Representational State Transfer)
- 2000년 **로이 필딩(Roy Fielding)**이 박사 논문에서 제안
- HTTP 프로토콜의 주요 설계자 중 한 명이 "웹의 구조를 API에도 적용하자"고 정리한 것
- 법적 표준이 아닌 **사실상의 표준(de facto standard)** — 업계 전체가 자발적으로 채택

### 핵심 구조
- **URL로 자원(데이터)을 지정** + **HTTP 메서드로 동작을 구분**

### HTTP 메서드 4가지
| 메서드 | 동작 | 식당 비유 | 이번 프로젝트 예시 |
|--------|------|-----------|-------------------|
| **GET** | 조회 | "메뉴판 보여줘" | 공공데이터포털에서 복지서비스 목록 조회 |
| **POST** | 생성 | "새 주문 넣어줘" | Gemini/Groq API에 질문 보내서 답변 생성 |
| **PUT** | 수정 | "3번 주문 변경해줘" | (이번 프로젝트에서는 미사용) |
| **DELETE** | 삭제 | "3번 주문 취소해줘" | (이번 프로젝트에서는 미사용) |

### REST, RESTful, REST API 관계
- **REST** — 설계 원칙 (아키텍처 스타일)
- **RESTful** — REST 원칙을 잘 따르고 있다는 형용사
- **REST API** — REST 원칙에 따라 만든 API

## 3. FastAPI

### FastAPI란?
- REST API를 Python으로 **빠르고 쉽게 만들 수 있게 도와주는 프레임워크(도구)**
- REST API가 "요리 레시피"라면, FastAPI는 "좋은 주방 도구"

### 언어별 REST API 프레임워크
| 언어 | 프레임워크 | 특징 |
|------|-----------|------|
| **Python** | FastAPI | 빠르고 현대적 (위니쿠키에서 사용) |
| **Python** | Flask | 가볍고 단순, 입문용 |
| **Python** | Django REST Framework | 대규모 서비스에 강함 |
| **Java** | Spring Boot | 기업용 서비스 압도적 1위 |
| **JavaScript** | Express.js | Node.js에서 가장 대중적 |
| **Go** | Gin | 성능 매우 빠름, 대규모 트래픽 |

→ 언어와 도구는 달라도 **REST라는 규칙은 동일**. 하나를 이해하면 다른 것도 파악 가능.

## 4. 개발 환경 세팅 완료 사항

### conda 환경
- 환경 이름: `public-chatbot`
- Python: 3.11
- 설치된 패키지: requests, python-dotenv, jupyter

### 프로젝트 구조
```
public-chatbot/
├── .env              # API 키 (GitHub에 올리지 않음)
├── .gitignore        # 민감 파일 제외 설정
├── environment.yml   # conda 환경 export (GitHub에 올림)
└── README.md
```

### .env 파일 구성
```
DATA_GO_KR_API_KEY=공공데이터포털_Decoding_키
GEMINI_API_KEY=Gemini_API키
GROQ_API_KEY=Groq_API키
```
→ 공공데이터포털은 **Decoding 키** 사용 (Encoding 키는 URL 중복 인코딩 문제 발생 가능)

### 회사-집 환경 동기화 방법
```bash
# 집에서 환경 변경 후
conda env export > environment.yml
git add . && git commit -m "환경 업데이트" && git push

# 회사에서 동기화 (최초)
git clone https://github.com/dino92-ys/public-chatbot.git
conda env create -f environment.yml

# 회사에서 동기화 (이후 업데이트)
git pull
conda env update -f environment.yml
```
→ 각 노트북마다 **.env 파일은 따로 생성** 필요 (GitHub에 안 올라가므로)

### Docker
- 지금은 사용하지 않음
- 배포 단계(Phase 5)에서 도입 예정
- "클라우드 배포를 위해 Docker를 적용했다"는 포트폴리오 어필 포인트가 됨

## 5. 핵심 교훈

1. **API = 추상화** — 내부를 몰라도 규칙만 지키면 사용 가능
2. **REST API = 웹의 방식을 그대로 사용** — URL + HTTP 메서드 조합
3. **프레임워크는 도구일 뿐** — 규칙(REST)을 이해하면 어떤 도구든 적용 가능
4. **환경 분리** — 프로젝트별 conda 환경 분리가 관리에 유리
5. **보안** — API 키는 반드시 .env + .gitignore 처리
